## Подготовка

Выполни параллельно, чтобы понять контекст:

```bash
gh pr view --json number,title,headRefName,state,url   # данные PR
git log main..HEAD --oneline                            # коммиты ветки
git diff main --stat                                    # затронутые файлы
```

Убедись, что PR существует и находится в состоянии `OPEN`. Если PR не найден — сообщи пользователю и остановись.

---

## Шаг 1 — Автоматические проверки

Запусти последовательно. При любой ошибке **остановись**, выведи вывод команды и попроси исправить перед мержем.

### 1.1 Формат ветки

Проверь, что имя текущей ветки соответствует шаблону `<prefix>/<kebab-name>`, где prefix — один из: `feat`, `fix`, `refactor`, `docs`, `chore`. Если нет — сообщи об ошибке.

### 1.2 Формат коммитов

Проверь каждый коммит ветки (`git log main..HEAD --format="%s"`). Каждый должен соответствовать:

```
<type>(<scope>): описание
```

где type — `feat|fix|refactor|docs|chore|test|style`. Если хотя бы один коммит не соответствует — выведи список нарушений.

### 1.3 Lint

```bash
npm run lint
```

### 1.4 Build

```bash
npm run build
```

## Шаг 2 — Чеклист ревью

После автоматических проверок выведи чеклист и **попроси пользователя подтвердить**, что пункты, помеченные `*`, выполнены. Пункты без `*` — автоматически не проверяемы и остаются на совести ревьюера.

Выведи чеклист в таком формате — отмечай `[x]` пункты, которые прошли автоматически, `[ ]` — требующие ручного подтверждения:

### Общее

- `[x]` Ветка создана от `main`, название по шаблону _(проверено в шаге 1.1)_
- `[x]` Коммиты по Conventional Commits _(проверено в шаге 1.2)_
- `[ ]` Один PR — одна логическая единица изменений
- `[ ]` Заголовок PR по Conventional Commits; body содержит **Что реализовано**, **Тест-план**
- `[x]` `npm run lint` проходит _(проверено в шаге 1.3)_
- `[x]` `npm run build` проходит _(проверено в шаге 1.4)_

### TypeScript

- `[ ]` Нет `any` без обоснования; нет `eslint-disable` без причины
- `[ ]` Нет дублирования типов из `@my-pocket/shared-types`
- `[ ]` `Decimal`-поля Prisma представлены как `string` в shared-types, не кастуются к `number` без `parseFloat`

### Shared-types

- `[ ]` При изменении API-контракта правки в бэке, фронте и `shared-types` — в одном PR
- `[ ]` Нет локального дублирования DTO/типов на фронте

### Backend

- `[ ]` Контроллер тонкий: только guards + commandBus/queryBus; бизнес-логика в handler
- `[ ]` Прямые вызовы `PrismaService` — только в `*.repository.ts`
- `[ ]` Новый модуль зарегистрирован в `AppModule`
- `[ ]` Все запросы к БД скоупированы по `userId`
- `[ ]` Роуты `summary`/`monthly` объявлены до `:id`
- `[ ]` Пароли хешируются через `bcrypt`; `JWT_SECRET` не хардкодится
- `[ ]` DTO с `class-validator`; глобальный `ValidationPipe` активен
- `[ ]` `P2002` → `ConflictException`, не 500
- `[ ]` После изменения `schema.prisma` — миграция создана и `prisma:generate` выполнен

### Frontend

- `[ ]` Импорты по правилу слоёв FSD: `app → views → widgets → features → entities → shared`
- `[ ]` Импорты только через публичный API среза (`index.ts`)
- `[ ]` `process.env.NEXT_PUBLIC_*` не используется напрямую — только через `env` из `shared/config`
- `[ ]` Query keys экспортируются из `entities/<name>/model`
- `[ ]` После мутаций инвалидируются транзакции + `summary` + `monthly`
- `[ ]` Все запросы через `apiClient`; нет прямого `fetch`/`axios` в компонентах
- `[ ]` shadcn/ui-компоненты из `shared/ui`; `cn()` для условных классов; `formatMoney` для сумм
- `[ ]` Ошибки мутаций не дублируются — глобальный `MutationCache` обрабатывает тосты

### Ручное тестирование

- `[ ]` Золотой путь пройден: логин → категория → транзакция → дашборд
- `[ ]` Проверен хотя бы один edge-case (токен / валидация / удаление)
- `[ ]` Нет регрессий в уже работавших фичах

## Шаг 3 — Подтверждение

Спроси пользователя:

> Все ручные пункты чеклиста проверены? Подтверди мерж (`да` / `нет`).

Если пользователь ответил `нет` или указал замечания — **не мержи**, выведи список что нужно исправить.

## Шаг 4 — Мерж

После подтверждения выполни:

```bash
gh pr merge <number> --merge --delete-branch
```

Флаг `--merge` создаёт merge-коммит (сохраняет историю веток). `--delete-branch` удаляет ветку после мержа.

Переключись на `main` и подтяни изменения:

```bash
git checkout main
git pull
```

Выведи итоговое сообщение: заголовок смерженного PR и ссылку на него.
